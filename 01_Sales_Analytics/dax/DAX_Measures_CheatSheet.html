MEASURE 01
Total Sales
Calculates the grand total of all revenue across the fact_Sales table. This is the foundation measure — almost every other measure references this one. It automatically respects whatever filter context is applied through slicers or visual interactions.
💼Business use: The primary KPI on every executive dashboard. Shows total revenue for the selected period, region, or category.
Format$ Currency
FunctionSUM()
Tablefact_Sales
ColumnSales
DifficultyBeginner
DAX
Total Sales = SUM(fact_Sales[Sales])
🎤 Interview Q&A
QWhy did you use SUM() and not SUMX() for Total Sales?
SUM() is used when you want to aggregate a single existing column directly. It is simpler and more performant for straightforward aggregations. SUMX() is needed when you want to perform a row-by-row calculation before aggregating — for example, multiplying two columns together per row. Since Sales is already a pre-calculated revenue column in our fact table, SUM() is the correct and more efficient choice.
QHow does Total Sales behave when a slicer is applied?
Total Sales automatically respects filter context. When a user selects Year = 2016 in a slicer, the measure evaluates only the rows where the year is 2016, returning the 2016 total. This is because DAX measures are always evaluated within the current filter context — they do not need explicit WHERE clauses like SQL.

💼Business use: Profitability analysis. Negative values in this measure signal problematic orders or categories that need attention.

MEASURE 03
Total Orders
Counts the number of UNIQUE orders. Uses DISTINCTCOUNT instead of COUNT because one Order ID can appear multiple times in the fact table — once per product line item in that order. DISTINCTCOUNT ensures each order is counted exactly once regardless of how many products it contains.
💼Business use: Volume metric showing how many orders were placed. Used in Average Order Value calculation.
FormatWhole Number
FunctionDISTINCTCOUNT()
Tablefact_Sales
ColumnOrder ID
DifficultyBeginner

🎤 Interview Q&A
QWhy DISTINCTCOUNT instead of COUNT or COUNTROWS?
COUNT counts non-blank values in a column — it would return 9,994 (total rows) which is wrong because the same Order ID appears multiple times for multi-product orders. COUNTROWS counts all rows in a table — same problem. DISTINCTCOUNT returns the count of unique values only, giving 5,009 unique orders which is the correct business metric.
QWhat is the difference between COUNT, COUNTA, COUNTROWS, and DISTINCTCOUNT?
COUNT counts numeric non-blank values. COUNTA counts all non-blank values including text. COUNTROWS counts total rows in a table or filter result. DISTINCTCOUNT counts unique values only. For counting unique orders, customers, or products, DISTINCTCOUNT is always the correct choice.

MEASURE 05
Profit Margin %
Calculates the percentage of revenue that becomes profit. Uses DIVIDE() instead of the division operator to handle zero-division gracefully. Formatted as a percentage — the underlying value is stored as a decimal (0.1247) and displayed as 12.47%.
💼Business use: Profitability health indicator. Our Superstore shows ~12.47% which is medium margin — signals room for pricing or cost optimization.

🎤 Interview Q&A
QWhy use DIVIDE() instead of the / operator?
The division operator throws a divide-by-zero error when the denominator is zero, breaking the visual. DIVIDE() accepts a third parameter — the alternate result when the denominator is zero — returning 0 or BLANK instead of an error. This makes reports resilient to edge cases, which is essential in production environments.
QWhy format as percentage in Measure Tools instead of multiplying by 100 in DAX?
Multiplying by 100 changes the underlying value from decimal to integer, which breaks Time Intelligence and mathematical operations on the measure. The Format property in Measure Tools only changes the display — the stored value remains 0.1247, which is mathematically correct for all downstream calculations.
MEASURE 06
Average Order Value
Calculates the average revenue generated per order. Divides Total Sales by Total Orders — NOT using AVERAGE() which would give average per line item, not per order. This is a critical distinction that interviewers frequently test.
💼Business use: Customer spending behavior. A rising Average Order Value indicates customers are buying more per transaction — a positive business signal.
Format$ Currency
FunctionDIVIDE()

DifficultyIntermediate

🎤 Interview Q&A
QWhy not use AVERAGE(fact_Sales[Sales]) for Average Order Value?
AVERAGE(Sales) calculates the mean of individual line item values — not the average per order. One order CA-2016-152156 may have 3 line items worth $100, $200, and $300. AVERAGE would treat these as 3 separate values. Our measure correctly sums the entire order ($600) and divides by 1 order, giving the true average value per transaction.

QWhat is CALCULATE and why is it the most important DAX function?
CALCULATE is the only DAX function that can modify filter context. Every other function respects the existing filter context passively. CALCULATE actively changes it — it can add new filters, remove existing filters using ALL(), or replace filters entirely. Without CALCULATE, it would be impossible to build measures like percentage of total, period comparisons, or conditional aggregations.
MEASURE 08
Sales % of Total
Calculates each row's contribution as a percentage of the absolute grand total. Uses ALL(fact_Sales) to remove every filter from the denominator, ensuring the grand total is always $2.3M regardless of what filters are applied to the numerator.
💼Business use: Contribution analysis. Shows what percentage of total company revenue comes from each region, product, or customer.
FormatPercentage %

Pattern% of Total

Sales % of Total = 
DIVIDE(
    [Total Sales],
    CALCULATE([Total Sales], ALL(fact_Sales)),

)
🎤 Interview Q&A
QWhy does this measure show 100% on a Card visual with no filters?
When no filters are applied, the numerator and denominator both evaluate to the same grand total — $2.3M / $2.3M = 100%. This is mathematically correct. The measure is designed for table or matrix visuals where each row has its own filter context — there it correctly shows each row's percentage contribution to the grand total.
QWhat does ALL() do in this context?
ALL() removes all filters from the specified table or column. In this measure, ALL(fact_Sales) inside CALCULATE tells Power BI to ignore every filter — year, region, category — when calculating the denominator. This ensures the denominator always represents the true grand total, making percentage calculations accurate.

MEASURE 10
Sales MTD — Month to Date
Month to Date (MTD) accumulates sales from the first day of the current month up to the last date in the current filter context. Requires a properly marked Date Table to function. Resets to zero at the start of every new month.
💼Business use: "How much have we sold this month so far?" Used in weekly sales meetings to track monthly progress.
Format$ Currency
FunctionTOTALMTD()
RequiresMarked Date Table
DifficultyIntermediate
DAX
Sales MTD = 
TOTALMTD(
    [Total Sales],
    dim_Date[Date]
)
🎤 Interview Q&A
QWhat is the difference between MTD, QTD, and YTD?
MTD (Month to Date) accumulates from the 1st of the current month to the current date — resetting monthly. QTD (Quarter to Date) accumulates from the 1st of the current quarter — resetting every 3 months. YTD (Year to Date) accumulates from January 1st of the current year — resetting annually. All three require a marked Date Table and are commonly used together on executive dashboards.
QWhy does this measure return BLANK when no date filter is applied to current data?
TOTALMTD calculates for the current month in context. When no filter is applied, the current month defaults to the system date — May 2026 — which has no data in our 2014-2017 dataset. This is correct behavior. The measure works properly when a date slicer is used to select a period within the data range.

QWhat is the prerequisite for Time Intelligence functions to work in Power BI?
Three requirements must be met: First, a Date table must exist with a continuous date column covering the full data range with no gaps. QWhat is the difference between SAMEPERIODLASTYEAR and DATEADD with -1 YEAR?
SAMEPERIODLASTYEAR is purpose-built for year-over-year comparison and handles leap year edge cases automatically — for example, February 29 in a leap year maps to February 28 in a non-leap year. DATEADD(-1, YEAR) is a more general function that shifts any date by any number of periods. For annual comparisons, SAMEPERIODLASTYEAR is preferred for its clarity and automatic edge case handling.

QHow flexible is DATEADD and what parameters does it accept?
DATEADD accepts three parameters: the date column, the number of intervals to shift (positive or negative), and the interval type. Interval types are DAY, MONTH, QUARTER, or YEAR. MEASURE 19
Average Selling Price
Calculates the true average revenue per unit sold by using SUMX to compute total revenue row by row (Sales × Quantity), then dividing by total units. This is fundamentally different from AVERAGE(Sales) which ignores quantity weighting entirely.
💼Business use: Pricing analysis. If Average Selling Price drops over time, it signals pricing pressure, heavy discounting, or a shift toward cheaper product mix.
Format$ Currency
FunctionSUMX() DIVIDE()
PatternWeighted Average
DifficultyAdvanced
DAX
Average Selling Price = 
DIVIDE(
    SUMX(
        fact_Sales,
        fact_Sales[Sales] * fact_Sales[Quantity]
    ),
    SUM(fact_Sales[Quantity]),
    0
)
🎤 Interview Q&A
QExplain the difference between SUM and SUMX with a real example.
SUM aggregates an entire column in one operation — SUM(Sales) adds all values in the Sales column. SUMX iterates row by row, evaluates an expression per row, then sums the results. For example: if Row 1 has Sales=$100 and Quantity=3, and Row 2 has Sales=$200 and Quantity=1 — SUM(Sales) = $300, but SUMX(fact_Sales, Sales * Quantity) = ($100×3) + ($200×1) = $500 which is the actual total revenue. SUMX is essential when the calculation requires combining multiple columns per row.
QWhat is the naming pattern for iterator functions in DAX?
All iterator functions end with the letter X: SUMX, AVERAGEX, COUNTX, MAXX, MINX, RANKX, PRODUCTX. They all share the same two-parameter pattern — the table to iterate over, and the expression to evaluate per row. The X suffix is a reliable signal that the function operates row by row rather than on an entire column at once.
MEASURE 20
Avg Discount %
Calculates the average discount percentage applied across all orders using AVERAGEX. While AVERAGE(Discount) would give the same result here, AVERAGEX provides greater flexibility — it can be applied to a filtered subset of rows, making it extensible for conditional discount analysis.
💼Business use: Discount management. High average discounts erode profit margins. Used alongside Profit Margin % to understand the discount-profit relationship.
FormatPercentage %
FunctionAVERAGEX()
PatternRow Iterator
DifficultyIntermediate
DAX

AVERAGEX(
    fact_Sales,

🎤 Interview Q&A
QWhen would AVERAGEX give a different result from AVERAGE?
They give the same result when applied to a raw column. AVERAGEX becomes essential when the expression involves a calculation rather than a single column — for example, AVERAGEX(fact_Sales, fact_Sales[Sales] / fact_Sales[Quantity]) calculates average price per unit row by row before averaging, which is impossible with AVERAGE alone. Also, AVERAGEX can be applied to a FILTER result — AVERAGEX(FILTER(fact_Sales, [Profit] > 0), [Discount]) gives the average discount only on profitable orders.

Product Rank
Ranks each product by Total Sales using RANKX with DENSE ranking to avoid gaps when products have equal sales. ALL(dim_Product) ensures every product is considered in the ranking, not just the filtered subset visible in the current visual.
💼Business use: Product performance league table. Identifies top sellers and underperformers. Changes dynamically when date or region filters are applied.
FormatWhole Number
FunctionRANKX() ALL()
OrderDESC (High = Rank 1)
DifficultyAdvanced
DAX
Product Rank = 
RANKX(
    ALL(dim_Product),
    [Total Sales],
    ,
    DESC,
    DENSE
)
🎤 Interview Q&A
QExplain all 5 parameters of RANKX.
Parameter 1 is the table to rank across — ALL(dim_Product) removes filters so all products are ranked. Parameter 2 is the value expression to rank by — [Total Sales]. Parameter 3 is the value expression for the current row — left blank to use the default current context value. Parameter 4 is the order: DESC means highest sales gets Rank 1, ASC means lowest gets Rank 1. Parameter 5 is the tie-breaking method: DENSE means tied values share a rank with no gap (1,1,2,3), SKIP means tied values share a rank but leave a gap (1,1,3,4).
QWhy is ALL(dim_Product) necessary in RANKX?
Without ALL(), RANKX would only consider the products currently visible in the filter context. If a visual is filtered to show only Furniture products, every Furniture product would rank against each other — the top Furniture product would be Rank 1. With ALL(dim_Product), every product in the entire dataset is considered, so Rank 1 is truly the best-selling product across all categories.

VAR / RETURN PATTERN

CATEGORY 05 · 2 MEASURES
MEASURE 24
Sales Performance Status
Uses VAR/RETURN to store intermediate calculations in named variables, then SWITCH(TRUE()) to categorize performance into four tiers based on Year over Year (YoY) Growth percentage. Demonstrates production-quality DAX code writing practices.
💼Business use: At-a-glance performance indicator. Executives see colored status labels instead of raw numbers — enabling faster decisions.
FormatText
FunctionsVAR RETURN SWITCH DIVIDE
PatternVAR/RETURN + SWITCH
DifficultyAdvanced
DAX
VAR _CurrentSales  = [Total Sales]
VAR _PreviousSales = [Sales PY]
VAR _GrowthPct = 
    DIVIDE(
        _CurrentSales - _PreviousSales,
        _PreviousSales,
        0
    )
RETURN
    SWITCH(
        TRUE(),
        _GrowthPct >= 0.1,   "🟢 Excellent (>10%)",
        _GrowthPct >= 0,     "✅ Growing",
        _GrowthPct >= -0.1,  "⚠️ Declining",
                               "🔴 Critical (>10% drop)"
    )
🎤 Interview Q&A
QWhat are the advantages of using VAR/RETURN over writing the full expression inline?
Three key advantages: First, performance — variables are calculated only once regardless of how many times they are referenced in the RETURN block. Without VAR, [Sales PY] would be calculated twice — once in the numerator subtraction and once in the denominator division. Second, readability — named variables make the intent of each calculation self-documenting. Third, debuggability — you can temporarily change RETURN to return an intermediate variable (RETURN _GrowthPct) to inspect values during development.
QHow does SWITCH(TRUE()) work and why is it better than nested IF statements?
SWITCH(TRUE()) evaluates each condition in order and returns the result of the first condition that evaluates to TRUE. It stops checking once a match is found. Compared to nested IF statements, SWITCH(TRUE()) is more readable, easier to maintain, and more performant for multiple conditions. Adding a new tier requires one line instead of restructuring the entire nested IF chain.
MEASURE 25

Classifies the current context into one of four profit margin categories using VAR to store the Profit Margin % and SWITCH(TRUE()) for clean multi-condition logic. Demonstrates the same pattern applied to profitability instead of growth.
💼Business use: Margin health flagging. Quickly identify which products, regions, or customer segments are loss-making versus high-margin.

    )
🎤 Interview Q&A
QCan VAR store a measure reference and if so why is that useful?
Yes — VAR can store the result of any DAX expression including measure references. VAR _Margin = [Profit Margin %] evaluates the measure once and stores the decimal result. This is particularly useful when the same measure is referenced multiple times in different conditions — the measure calculates once and the stored value is reused, which is significantly more performant than recalculating it for each condition check.

MEASURE 26
Running Total Sales
Calculates a cumulative sum of sales that keeps growing over time. Uses CALCULATE with a FILTER on ALL(dim_Date) to include all dates up to and including the current date in context. Unlike YTD (Year to Date) which resets yearly, Running Total never resets — it accumulates from the very beginning of the data.
💼Business use: Cumulative growth visualization. Used in waterfall charts and area charts to show the total business built over time. Revenue milestones are tracked with this measure.
Format$ Currency
FunctionsCALCULATE FILTER ALL MAX
PatternCumulative Calculation
DifficultyAdvanced
DAX
Running Total Sales = 
CALCULATE(
    [Total Sales],
    FILTER(
        ALL(dim_Date),
        dim_Date[Date] <= MAX(dim_Date[Date])
    )
)
🎤 Interview Q&A
QExplain how the Running Total logic works step by step.
Step 1: ALL(dim_Date) removes all existing date filters, making every date available for evaluation. Step 2: FILTER then applies a new condition — keep only dates where dim_Date[Date] is less than or equal to MAX(dim_Date[Date]). Step 3: MAX(dim_Date[Date]) returns the last date in the current row's context — for example, if the visual shows February 2016, MAX returns February 29, 2016. Step 4: The filter keeps all dates from the beginning of the dataset up to February 29, 2016. Step 5: CALCULATE evaluates [Total Sales] in this filtered context — returning cumulative sales from day 1 through February 2016.
QWhat is the difference between Running Total and YTD (Year to Date)?
YTD resets to zero at the start of every year — on January 1, 2016 it starts counting again from zero. Running Total never resets — it accumulates from the very first date in the dataset continuously. YTD is useful for tracking annual targets. Running Total is useful for showing absolute business growth over the entire history of the company.
MEASURE 27
Dynamic Title
Creates a context-aware text string that changes based on what the user has selected in the Year slicer. Uses SELECTEDVALUE which returns the selected value when exactly one item is chosen, and falls back to a default text when no selection or multiple selections exist. The & operator concatenates text strings.
💼Business use: Professional report UX. When a user selects 2016, the report title automatically reads "Sales Analysis — 2016" making the report feel interactive and eliminating confusion about which period is being viewed.
FormatText
FunctionSELECTEDVALUE() &
PatternDynamic UX
DifficultyIntermediate
DAX
Dynamic Title = 
"Sales Analysis — " & 
SELECTEDVALUE(
    dim_Date[Year],
    "All Years"
)
🎤 Interview Q&A
QHow does SELECTEDVALUE work and what is the second parameter for?
SELECTEDVALUE returns the value of a column when exactly one distinct value is in the filter context. The second parameter is the alternate result — returned when zero values or more than one value is selected. Without the second parameter, SELECTEDVALUE returns BLANK when no single item is selected, which would produce "Sales Analysis — " with nothing after the dash. The alternate result "All Years" makes the title informative in all scenarios.
QHow is this measure used in a visual?
The Dynamic Title measure is used in the Title property of a visual's formatting options. In the visual's Format pane, go to Title, enable it, then in the Title text field click the fx (conditional formatting) button and select the Dynamic Title measure as the value. The visual title will now automatically update whenever the year slicer selection changes — without any manual intervention.
QHow would you extend this to show both the selected year and region?
Extend the concatenation: "Sales Analysis — " & SELECTEDVALUE(dim_Date[Year], "All Years") & " | " & SELECTEDVALUE(dim_Region[Region], "All Regions"). This produces titles like "Sales Analysis — 2016 | West" when both slicers have single selections, and gracefully falls back to "All Years" or "All Regions" when multiple items are selected.
